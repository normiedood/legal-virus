# Contributing to Legal Virus

Thanks for your interest in contributing! We appreciate all contributions — from documentation fixes to code and tests.

Before you begin
- Check existing issues and pull requests to avoid duplication.
- If you plan a large change, open an issue first to discuss the design.

Branching & workflow
1. Fork the repository.
2. Create a descriptive branch: `git checkout -b feat/my-feature`.
3. Make focused commits with clear messages. Use imperative tense (e.g., "Add feature X").
4. Rebase or merge the latest `main` before opening a pull request.

Coding standards
- Follow the existing coding style in the repository.
- Add or update tests for any new behavior.
- Keep changes focused and document any public API changes.

Commit messages
- Use short, descriptive titles and an optional body with additional context.
- Include a reference to an issue when relevant: `Fixes #123`.

Pull requests
- Open a PR against the `main` branch in this repository.
- Provide a clear description of what changed and why.
- Include testing instructions and the steps a reviewer should take.
- Link related issues and mention maintainers if you need a review.

Review process
- Maintainters and reviewers may request changes. Please address feedback promptly.
- Small documentation or typo fixes may be merged quickly.

Community & communication
- Be respectful and patient in reviews and discussions.
- For sensitive or security issues, contact the maintainers privately (see CODE_OF_CONDUCT.md for contact info).

Thank you for contributing to Legal Virus!
